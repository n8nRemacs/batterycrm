## ТЗ: Унифицированная выдача параметров обращения

### Цель
Бэкенд должен полностью управлять перечнем атрибутов, отображаемых на клиентских интерфейсах (Android/mini-app). Фронт отображает параметры “как есть” по заданному стандарту, без ручного описания каждого поля.

### Общие требования
- Каждый параметр описывается объектом `{ id, label, value, order }`.
- `id` — уникальный ключ (snake_case), стабильный между версиями.
- `label` — человекочитаемое название (уже локализованное).
- `value` — текстовое значение, подготовленное к выводу (например, “1 ч 30 мин” вместо “90”).
- `order` — число (можно дробное), определяющее позицию внутри секции; сортировка по возрастанию.
- Параметры с пустым `value` (null/пустая строка) не включаются в массив.
- Допускается группировка (например, `group`), но базовый стандарт для фронта — линейный список, отсортированный по `order`.

### API `GET /api/operator/appeals/:id`
1. В объекте `appeal` вернуть новое поле `meta`:
   ```json
   "appeal": {
     "...": "...",
     "meta": [
       { "id": "client_name", "label": "Клиент", "value": "РемАкс MD", "order": 100 },
       { "id": "parts_owner", "label": "Чья деталь", "value": "Наша", "order": 200 },
       { "id": "repair_type", "label": "Тип ремонта", "value": "Assist", "order": 210 }
     ]
   }
   ```
2. Наполнять `meta` всем набором параметров, которые должны видеть операторы: стадия, статус, источник, тип ремонта, владелец детали, оценки по стоимости/срокам, бренд/модель, режим обработки, и т.д.
3. `order` хранить в одном стиле (например, шаг 10) — это обеспечит предсказуемый порядок (100 — основной блок, 200 — информация об устройстве, 300+ — расчёты и т.д.).
4. Для дополнительных секций (клиент, устройство, история и пр.) использовать тот же формат либо отдельные массивы `client_meta`, `device_meta` по аналогии — фронт сможет отрисовать несколько блоков одним компонентом.
5. При отсутствии данных возвращать пустой массив `meta: []`, чтобы фронт не строил запасные сценарии.

### Валидация / тесты
- Обновить мок/тестовые данные (в т.ч. для `test@batterycrm.ru`) так, чтобы `meta` включало минимум 5 разных параметров и корректно задавало `order`.
- Добавить автоматическую проверку в n8n/скриптах: перед отправкой ответа фильтровать пустые значения, чтобы фронт не занимался очисткой.

### Сроки / зависимости
- После обновления API уведомить фронт, что массив `meta` готов и полностью покрывает отображение.
- До выката убедиться, что старые клиенты не упадут (поле опциональное, фронт обрабатывает отсутствие массива).
